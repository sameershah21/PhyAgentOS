# Physical Agent Operating System: 一种基于协议解耦与多智能体协同的自进化具身框架
**(Physical Agent Operating System: A Decoupled Protocol-Based Framework for Self-Evolving and Cross-Embodiment Agents)**

> **版本**: 0.1.0-Release | **发布日期**: 2026-03-20

## 摘要 (Abstract)
随着大语言模型（LLM）Agentic 能力的质变，具身智能（Embodied AI）领域正面临技术路线的关键分化。传统的端到端视觉-语言-动作（Vision-Language-Action, VLA）模型虽展现出端到端优化的潜力，但其“黑盒”特性导致了可解释性缺失、局部调试困难及跨本体泛化能力受限等根本性瓶颈。当前 VLA 模型常被质疑是依赖“视觉-运动先验”而非真正遵循语言指令，且在失败时缺乏原则性的诊断机制。为此，我们提出 Physical Agent Operating System（PhyAgentOS），一种基于 Agentic 工作流的自进化具身智能框架，通过引入“认知-物理解耦”的架构范式，实现了从强推理云端模型到边缘物理执行层的标准化映射。PhyAgentOS 不再追求端到端神经网络的单体优化，而是通过构建语言-动作接口（Language-Action Interface），将动作表示与本体形态彻底解耦。本框架以极轻量级的 `nanobot` 运行环境为底座，定义了 `TASK`, `SKILL`, `ACTION`, `ENVIRONMENT`, `EMBODIED` 等结构化协议层。该架构原生支持跨硬件平台的零代码迁移、基于沙盒的代码工具自生成，以及基于多智能体验证（Multi-Agent Critic）的安全纠偏机制。此外，PhyAgentOS 构建了统一的硬件抽象层（HAL），通过 `BaseDriver` 接口和动态加载机制，实现了对异构机器人本体（如机械臂、四足机器人、桌面宠物）的无缝支持。架构特性表明，PhyAgentOS 在保持 LLM 强推理能力的同时，具备 VLA 范式难以企及的可解释性、跨本体零样本迁移能力与系统级安全保证。

## 1. 引言 (Introduction)

赋予物理实体感知、推理并与真实世界交互的能力，是具身智能的核心目标。近年来，基于大规模异构数据联合训练的 VLA 模型实现了从像素到动作的端到端映射。然而，VLA 范式在实际部署中暴露了显著的系统性风险：
1. **不可解释性与调试困难**：决策过程隐式分布在神经网络参数中，当机器人出现意外行为时，操作者缺乏可控的干预与诊断接口。
2. **执行幻觉与安全隐患**：在分布外（OOD）场景下，由于高维视觉特征向离散控制信号投影时的语义鸿沟，规划器容易忽视物理视觉证据，导致灾难性的物理碰撞。
3. **本体强耦合**：现有系统通常与特定机器人本体的运动学特性深度绑定。即使经过多本体预训练，在部署至全新构型的硬件时，仍需高昂的数据收集与微调成本。

基于上述挑战，本文提出：**在强推理大模型时代，具身控制的映射应通过标准化的结构化协议（Structured Protocols）而非隐式神经网络来实现。** 受到最新跨本体迁移研究的启发，PhyAgentOS 框架将具身任务分解为层次化的状态文件，云端 Agent 扮演“认知核心”的角色，通过读写标准协议来实现对物理世界的感知与控制。

本文的主要贡献如下：
*   提出了一种基于“认知-物理解耦”的具身智能架构，通过划分方法论（`SKILL`）与执行约束（`ACTION`），解决了跨本体泛化问题。
*   设计了基于 `nanobot` 的极轻量边缘运行时框架，支持从微控制器（如 ESP32）到工控机的高效部署与零代码迁移。
*   引入了沙盒驱动的代码生成机制与 Multi-Agent Critic 校验管线，实现了工具的自我进化与物理执行偏差的实时纠正。
*   构建了统一的硬件抽象层（HAL），通过 `BaseDriver` 接口和动态加载机制，实现了对异构机器人本体（如机械臂、四足机器人、桌面宠物）的无缝支持。

## 2. 架构设计：认知与物理的结构化解耦

PhyAgentOS 框架的核心在于确立了云端认知推理（Cognitive Reasoning）与边缘物理执行（Physical Execution）之间的明确边界。系统摒弃了直接的 API 强调用，转而采用一种**共享状态协议空间 (Shared State Protocol Space)**。

### 2.1 动态协同与认知状态空间
系统在运行时维护一组结构化的文本文件，用于表征全局状态。云端 Agent 仅对这些降维后的符号表示进行处理，避免了多模态底层特征的直接干扰：
*   **`TASK.md` (全局编排协议)**：作为系统的动态任务黑板（Blackboard Architecture）。用于对宏观指令进行有向无环图（DAG）级别的任务拆解，并维护多 Agent 或多本体间的时序依赖。
*   **`ENVIRONMENT.md` (环境感知表征)**：视觉-语言模型（VLM）及传感器的输出聚合端。将原始像素流降维为包含拓扑关系与三维坐标的结构化场景图（Scene-Graph），解决符号落地问题（Symbol Grounding）。
*   **`EMBODIED.md` (本体声明与运动学约束)**：边缘硬件向云端暴露的硬件抽象层接口。静态记录设备的自由度、通信协议、SDK 映射关系及物理极值（如最大负载、工作空间边界）。

### 2.2 逻辑解耦：`SKILL` 与 `ACTION`
为解决具身智能中动作与特定硬件强绑定的问题，PhyAgentOS 将传统意义上的“技能”拆分为抽象级与实例化级两个正交的维度：

*   **`SKILL.md` (系统级工作流 / 认知方法论)**：
    定义完成特定任务的通用状态机或操作管线，与具体物理参数无关。
    *例如在新设备零样本部署场景中，Agent 调用的“部署 `SKILL`”为：(1) 扫描端口网络信息 -> (2) 查询硬件型号与协议 -> (3) 检索对应 SDK 文档 -> (4) 编写并下发连通性测试代码。此流程对任何机械臂均通用。*
*   **`ACTION.md` (实例化的物理约束目标)**：
    表示在当前 `ENVIRONMENT` 与当前 `EMBODIED` 条件下，由 `SKILL` 实例化产生的具体物理执行参数。
    *例如在“抓取”任务中，Agent 检索旧有经验（如抓取橙子的 20N 闭合力），结合当前视觉反馈的目标尺寸与自身运动学限制，微调生成全新的 `ACTION.md`。底层求解器读取该约束文件以生成连续的轨迹序列（Trajectory Generation）。*

### 2.3 硬件抽象层 (HAL) 与看门狗机制 (Watchdog Mechanism)
为了支持异构机器人本体，PhyAgentOS 引入了统一的硬件抽象层（HAL）。在 PhyAgentOS 框架中，认知层（Track A，云端大模型）与物理执行层（Track B，边缘硬件）被严格物理隔离。这种隔离不仅体现在代码结构上，更体现在运行时的进程边界上。连接这两个异构世界的唯一桥梁，是基于文件系统的状态协议空间（Workspace API）以及负责轮询该空间的**硬件看门狗守护进程 (HAL Watchdog Daemon)**。

`hal_watchdog.py` 并非简单的脚本执行器，而是整个物理执行层的核心调度器与状态同步器。其主要职责包括：

1.  **动态驱动加载与本体实例化 (Dynamic Driver Loading)**：
    看门狗在启动时，通过命令行参数（如 `--driver go2_edu` 或 `--driver simulation`）从驱动注册表（Driver Registry）中动态加载对应的 `BaseDriver` 实现。这一机制使得同一套 PhyAgentOS 认知系统可以无缝对接从桌面宠物到工业双臂机器人的各类异构硬件。
2.  **物理先验知识的注入 (Profile Injection)**：
    加载驱动后，看门狗会自动将该硬件的本体声明文件（`EMBODIED.md`，包含自由度、运动学极限、支持的动作空间等）复制到共享工作区。这相当于在系统启动时，向云端认知层注入了当前物理实体的“自我认知（Self-Awareness）”。
3.  **异步指令轮询与执行 (Asynchronous Command Polling)**：
    看门狗以设定的频率（如 1Hz）持续轮询工作区中的 `ACTION.md` 文件。当检测到由 Critic Agent 校验通过并写入的新动作指令时，看门狗会解析该 JSON 格式的指令，并调用底层驱动的 `execute_action` 方法将其转化为具体的硬件控制信号（如 CAN 总线指令或 ROS2 消息）。
4.  **物理状态的闭环回写 (Closed-Loop State Synchronization)**：
    动作执行完毕后，看门狗会调用驱动的 `get_scene` 方法获取最新的物理世界状态（如机器人位姿、被操作物体的新坐标），并将其格式化为场景图（Scene-Graph）覆写回 `ENVIRONMENT.md`。随后，清空 `ACTION.md` 以向认知层发出“动作已完成”的信号，从而完成一个完整的“感知-规划-执行-反馈”闭环。

引入独立的看门狗进程带来了显著的架构优势：
*   **极高的系统鲁棒性 (Robustness)**：认知层（大模型推理）和执行层（硬件控制）的崩溃互不影响。即使云端网络断开或大模型服务异常，底层的看门狗和硬件驱动依然保持运行，确保机器人处于安全状态（如触发急停）。
*   **严格的时序解耦 (Temporal Decoupling)**：大模型的推理通常具有高延迟且不可预测，而底层硬件控制需要高频实时的响应。看门狗通过异步轮询文件系统，完美地缓冲了这种时序上的不匹配。
*   **透明的可观测性 (Observability)**：所有的控制指令和状态反馈都以纯文本（Markdown/JSON）的形式暴露在文件系统中。开发者可以随时打开 `ACTION.md` 或 `ENVIRONMENT.md` 查看系统当前的状态，极大地降低了调试具身智能系统的门槛。

## 3. 基于约束求解的具身控制范式 (Constraint-Based Embodied Control Paradigm)

当前主流的具身智能控制方法主要分为两类：一是基于强化学习（RL）或模仿学习（IL）的端到端策略（End-to-End Policies），直接从视觉输入映射到关节力矩；二是基于传统机器人学的精确轨迹规划（Trajectory Planning）。前者面临泛化性差和不可解释的“黑盒”问题，后者则难以应对非结构化环境中的语义任务。

PhyAgentOS 框架倡导并实践了第三种路径：**基于约束求解的具身控制范式 (Constraint-Based Control Paradigm)**。这一范式深受 ReKep (Relational Keypoint Constraints) 等前沿研究的启发，但 PhyAgentOS 关注的并非特定的算法实现，而是将其提升为一种系统级的架构设计原则。

### 3.1 语义意图与几何约束的映射

在 PhyAgentOS 中，大语言模型（LLM）不直接输出底层的关节角度或末端位姿序列。相反，LLM 的输出被限制在**语义意图（Semantic Intent）**和**几何约束（Geometric Constraints）**的层面。

当执行一个复杂的物理操作（如“将杯子倒空”）时，认知层（Track A）的工作流如下：
1.  **语义理解与任务拆解**：Planner Agent 理解“倒空”的语义，并将其拆解为“抓取杯子”、“移动到目标容器上方”、“倾斜杯子”等子任务。
2.  **关键点与关系约束生成**：对于“倾斜杯子”这一动作，Planner 并不计算具体的运动轨迹，而是生成一组空间关系约束。例如：
    *   约束 1：杯子底部关键点（Keypoint A）的 Z 坐标必须高于杯口关键点（Keypoint B）。
    *   约束 2：杯口关键点（Keypoint B）必须保持在目标容器开口（Keypoint C）的二维投影范围内。
    *   约束 3：末端执行器的姿态必须避免与桌面发生碰撞。
3.  **约束协议下发**：这些约束被格式化为结构化的 JSON 数据，写入 `ACTION.md`。

### 3.2 边缘侧的实时优化求解

物理执行层（Track B）接收到 `ACTION.md` 中的约束描述后，底层的约束求解器（Constraint Solver，如基于二次规划 QP 的优化器）接管控制权。

求解器结合当前机器人的运动学模型（Kinematics）、动力学限制（Dynamics Limits，定义在 `EMBODIED.md` 中）以及实时的传感器反馈，将这些高维的几何约束转化为一个最优化问题。求解器在极短的时间内（通常 < 50ms）计算出满足所有约束条件的最优关节轨迹，并驱动电机执行。

### 3.3 约束求解范式的核心优势

将控制范式从“轨迹生成”转变为“约束求解”，为 PhyAgentOS 带来了革命性的优势：

1.  **真正的跨本体泛化 (Cross-Embodiment Generalization)**：
    几何约束（如“保持 A 在 B 上方”）是与机器人本体无关的（Body-Agnostic）。同一个约束描述，可以被四轴机械臂、七轴协作臂甚至双臂机器人读取。底层的求解器会根据各自的运动学结构，计算出截然不同的关节运动，但最终达成相同的物理效果。这使得 PhyAgentOS 中的 `SKILL.md`（技能库）可以实现真正的零样本跨硬件迁移。
2.  **对环境扰动的强鲁棒性 (Robustness to Perturbations)**：
    由于底层求解器是实时运行的，当环境发生动态变化（如目标容器被移动）时，只要约束条件依然成立，求解器就能实时调整轨迹以适应变化，而无需云端大模型重新进行耗时的推理规划。
3.  **可解释性与安全边界 (Interpretability and Safety)**：
    约束条件是显式且人类可读的。如果机器人执行失败，开发者可以清晰地看到是哪一个约束（如防碰撞约束或关节极限约束）导致了无解。同时，硬性的物理约束（如“末端 Z 坐标必须 > 0”）在求解器层面保证了绝对的安全性，彻底杜绝了端到端模型可能产生的灾难性动作。

## 4. 跨本体泛化与超轻量边缘部署

### 4.1 `nanobot` 边缘运行时框架
现有具身框架（如传统 ROS 架构）往往对边缘侧算力有较高要求。PhyAgentOS 通过将繁重的推理与多模态解析任务卸载至云端 LLM API，在本地抽象出了极致轻量的 `nanobot` 框架。
`nanobot` 并非特定硬件实体，而是一个通用的 Agent 运行时环境（Runtime Environment）。该环境仅负责网络通信、协议解析与底层硬件 SDK 的路由调用。其极低的内存与算力开销，使得框架可部署于资源严格受限的微控制器（如 ESP32）或常规工控机中。

### 4.2 零代码跨本体平移 (Zero-Code Transfer)
在 PhyAgentOS 架构下，跨本体迁移转化为一个**基于文档约束的重新参数化过程**。
当一套在“越疆（Dobot）四轴机械臂”上验证通过的认知逻辑转移至“Franka 七轴协作机械臂”时：
1. `nanobot` 框架自动侦测并向云端上传 Franka 的 `EMBODIED.md`。
2. 云端 Agent 读取并更新当前的物理先验知识。
3. `SKILL.md` 维持不变，Agent 在生成 `ACTION.md` 时，依据新的物理极值（如臂展与自由度）自主重新约束求解空间。
此机制无需修改任何顶层控制逻辑代码，实现了真正的零样本硬件迁移。

## 5. 自进化机制：沙盒驱动的工具抽象

传统框架严格限制 LLM 编写底层控制代码以规避不可控的系统崩溃。PhyAgentOS 采用了一种“控制权分层”的策略：允许大模型进行代码生成，但其目的在于**构建与封装可复用的工具（Tool Creation）**，而非直接进行实时闭环控制。

### 5.1 授人以渔的沙盒演进 (Sandbox Evolution Pipeline)
面对超出既有动作空间（Action Space）的复杂未知任务，PhyAgentOS 提供了一个安全的运行沙盒（Sandbox）。
1. **动态生成与仿真**：Agent 在沙盒内生成 Python/C++ 算法脚本，并在受限的物理环境或数字孪生仿真（如 PyBullet）中进行低速测试。
2. **工具标准化收编**：验证通过的脚本被抽象并封装为 MCP（Model Context Protocol）工具节点，注入系统工具库。
3. **工作流迭代**：在后续任务中，Agent 直接通过上下文调用该标准化工具。此机制构成了 PhyAgentOS 系统“遇到异常 -> 生成测试 -> 固化工具 -> 扩容能力”的自我进化闭环。

## 6. 安全保障：基于多智能体验证的物理纠偏

针对大模型在物理交互中固有的“执行幻觉（Execution Hallucination）”，PhyAgentOS 构建了一套基于 Multi-Agent 的强校验管线，以保障物理执行的绝对安全与状态一致性。

### 6.1 Planner-Critic 校验架构
在实际任务执行中，大模型常出现“执行确认偏误”（例如：抓取指令已发送，视觉闭环尚未确认，模型即认定目标已被抓取并规划高速移动）。PhyAgentOS 引入了独立于 Planner 的 **Critic Agent**：
*   **状态强拦截**：Critic Agent 拥有底层传感器状态流（如夹爪开合度、六维力矩）的最高读写权限。
*   **时序中断与纠偏**：当 Planner 输出的执行状态与底层传感器数据不符时（例如 `Gripper_Width == 0` 且 `Force == 0`），Critic 立即抛出物理异常中断，强制 Planner 记录当前偏差并重新调用视觉接口校准环境。
*   **动作合法性校验**：通过 `EmbodiedActionTool`，Critic 在动作写入 `ACTION.md` 前，严格对照 `EMBODIED.md` 中的物理极限（如工作空间边界、最大负载）进行合法性校验。

### 6.2 闭环试错与长程经验固化
纠偏机制产生的价值不仅在于避免单次物理碰撞，更在于形成系统的**长程认知记忆（Epistemic Memory）**。
拦截日志、环境反馈以及设备文档解析结果，均被提炼为半结构化的自然语言经验，永久追加至 `KNOWLEDGE.md`（先验知识库）与 `LESSONS.md`（异常避坑指南）中。在后续周期的规划前置阶段，Agent 被强制要求检索这些记忆库，有效消除在相同物理环境下的重复失效，使系统表现出近似人类的经验积累特征。

## 7. 语义导航与多模态感知融合 (Semantic Navigation and Multimodal Perception)

为了实现更高级的自主性，PhyAgentOS 引入了语义导航和多模态感知融合机制，使机器人能够理解复杂的环境语义并执行高级导航指令。

### 7.1 语义导航工具 (Semantic Navigation Tool)
PhyAgentOS 提供了 `SemanticNavigationTool`，允许 Agent 将高层语义目标（如“移动到厨房的桌子旁”）解析为具体的物理坐标。
*   **目标解析**：工具读取 `ENVIRONMENT.md` 中的场景图（Scene-Graph），根据目标类别（`target_class`）、节点 ID（`target_id`）或区域名称（`zone_name`）定位目标。
*   **位姿计算**：结合机器人当前位姿和目标位置，计算出合适的接近位姿（Approach Pose），并生成底层的导航动作指令。

### 7.2 多模态感知管线 (Perception Pipeline)
PhyAgentOS 的感知服务（`PerceptionService`）负责将原始传感器数据转化为结构化的场景图：
*   **几何处理 (Geometry Pipeline)**：处理点云和里程计数据，提取环境的几何特征和障碍物信息。
*   **语义分割 (Segmentation Pipeline)**：利用视觉模型对图像进行语义分割，识别环境中的物体类别。
*   **数据融合 (Fusion Pipeline)**：将几何特征与语义信息融合，构建包含物体三维坐标、类别和拓扑关系的场景图，并由 `EnvironmentWriter` 写入 `ENVIRONMENT.md`。

## 8. 结论 (Conclusion)

本文提出了 Physical Agent Operating System (PhyAgentOS)，一种旨在克服端到端 VLA 模型不可解释性与泛化瓶颈的创新具身架构。通过设计 `nanobot` 边缘底座与一套由 `TASK`, `SKILL`, `ACTION` 等构成的认知-物理状态协议，PhyAgentOS 成功将复杂的机器人控制降维为结构化文档的读写过程。

系统架构分析表明，该框架不仅实现了极其轻量化的边缘部署与跨本体的零代码平移，更通过多智能体 Critic 校验、沙盒工具生成机制以及统一的硬件抽象层（HAL），赋予了具身智能体高度的安全约束与可持续进化的能力。PhyAgentOS 为构建去本体化、可解释且高度可复用的下一代通用具身智能底座提供了一种极具潜力的范式转移路线。
